# Dryrun — 2026-05-26 (Claude Code self-dogfood)

Real-use dryrun of the post-2026-05-26 shhh build (per-project install
picker + positional CLI + semantic diff renderer + `shhh audit .` scope
filter). The agent running this dryrun is itself a Claude Code session
with shhh hooked globally (`~/.local/bin/shhh`, freshly copied from
`bin/shhh`). Every Read/Bash call I make during this session passes
through shhh — so any rough edge I hit is, by definition, a rough edge
a real user will hit.

Goal: spot anything that frustrates the forcing-function scenario
(`shhh install claude-code` → `claude` → leaks redacted) before more
features land on top.

Findings are logged inline; the "Summary" section at the bottom is
what I'll triage with the maintainer at the end.

## Findings

### F1 — CRITICAL — `--no-serve` (and any flag) silently ignored when written after a positional arg

**Repro:**
```
shhh audit . --no-select --no-serve
```
Expected: terminal output only, no HTTP server, clean exit.
Actual: full audit ran, CLI summary printed, **then started the HTTP server**
and blocked on Ctrl-C. Confirmed with a debug print: `noServe=false` despite
the user passing `--no-serve`.

**Root cause:** Go's stdlib `flag.Parse` stops at the first non-flag
argument. After `.` was seen as a positional, every subsequent
`--flag` was treated as another positional, not as a flag. This is
documented stdlib behavior but ruthlessly counter-intuitive — users
expect flag order to be irrelevant (every other CLI in their toolbox
allows interleaving).

**Same bug class in `cmdinstall`:** `shhh install claude-code /tmp/x --scope global`
silently treated `--scope` and `global` as TWO extra positional paths
and happily created `.claude/` directories named `--scope/` and `global/`
in the cwd. Verified with stray dirs `./shhh/--scope/` and `./shhh/global/`
during the dryrun (cleaned up).

**Fix landed:**
- `cmd/shhh/cmdaudit/cmdaudit.go::splitFlagsAndPositionals` — boolean-only
  split (all cmdaudit flags are bool, so trivial).
- `cmd/shhh/cmdinstall/cmdinstall.go::splitArgsByFlags` — value-aware
  split (consults the FlagSet to know which flags consume the next
  token; `--scope project` is a flag-value pair, `--verbose` is a
  standalone bool).
- Tests: `TestSplitFlagsAndPositionals` (6 cases) and
  `TestSplitArgsByFlags` (5 cases) lock the regression.

**Decision to revisit:** if cmdaudit ever adds a string flag, the
trivial bool-only split breaks. Long term, both packages should use
the value-aware variant from cmdinstall. Not refactored now to keep
the dryrun fix minimal.

### F2 — MEDIUM — install eagerly creates `.claude/` for any path string

Side-effect of F1: when the broken parser interpreted `--scope` as a
positional path, `installClaudeCode` happily created
`./shhh/--scope/.claude/` and `./shhh/global/.claude/` directories
without sanity-checking. There's no validation that the target path
"looks like a project root" (e.g., no `.git/`, no source files, weird
name starting with `--`).

With F1 fixed this can't be triggered the same way, but the install
code remains permissive. A user typing
`shhh install claude-code .claude` (typo for `--scope project .`)
would still create `./.claude/.claude/`.

**Suggested follow-up (not done):** refuse paths that:
- begin with `-` or `--` (obvious flag typo)
- already end in `.claude` (double-nesting)
And/or warn when the directory doesn't appear to be a project root
(no `.git/`, no `*.toml/*.json/*.mod/*.lock` near it).

### F3 — MEDIUM — `~/.shhh/config.json` out of sync with reality

`config.json::installed_paths` claimed `~/.claude/settings.json` was
hooked, but the actual file had `"hooks": {}` empty. The desync
likely came from a manual edit, an outside-of-shhh uninstall, or a
prior dryrun side-effect — not reproducible from the current code
path. But the audit's PROTECTED/NOT HOOKED decision is driven by
config.json, so a stale entry makes the audit lie ("you're hooked!"
when you're not).

The 2026-05-25 audit polish already added defensive re-reading of
each referenced settings.json (`internal/audit/run.go::settingsHasShhhHook`)
which catches this on the audit side — so the audit's status badges
are honest. But the config file itself stays wrong until the user
re-installs or manually edits it.

**Suggested follow-up (not done):** when `LoadConfig` reads
`installed_paths`, optionally verify each path still resolves to a
settings.json with a real shhh hook, and self-heal by dropping stale
entries (with a one-line warning to stderr). Cheap; saves the user
from a confusing future audit.

### F4 — KNOWN LIMITATION — hook activation requires session restart

I am Claude Code. When I reinstalled shhh mid-session, my own tool
calls did not start going through the hook — Claude Code reads
`settings.json` at session boot. So I could not validate the
redaction path (`read .env` → `[STRIPE_LIVE_KEY:...]` placeholder)
from inside this very session. The install command's footer already
says "Restart any running `claude` sessions for the hook to take
effect", so this is documented, but it's the single biggest
limitation of the self-dogfood pattern: **the dryrun agent cannot
verify the forcing-function scenario itself**.

The maintainer needs to:
1. Confirm the install / uninstall diff renderer feels right (done — user said "ça fonctionne au poil").
2. Open a fresh `claude` session in this repo (or any) and Read a
   fixture with fake secrets (e.g., `testdata/fixtures/leaky-project/.env`).
3. Confirm the placeholders appear in the transcript instead of raw
   values.

### F5 — LOW — `shhh audit .` reports 2 projects, not 1

Scope `.` from inside the shhh repo matches:
- `~/Documents/Musubi42/open-source/shhh` (the repo itself)
- `~/Documents/Musubi42/open-source/shhh/demo/leaktest` (descendant project)

Technically correct — the scope filter is recursive — but a user
running `shhh audit .` in their monorepo root may be surprised when
sub-projects also show up. Not a bug, but a UX note: the announcement
line `🛡️  shhh audit — scanning 2 projects, ~7 sessions` should
probably indicate that the count is "under this scope" so it doesn't
read as "you have 2 projects total."

**Suggested wording:** `🛡️  shhh audit — scope ./, 2 projects (~7 sessions)`

---

### F6 — LOW — full-tree enumeration before scope filter

`internal/audit/run.go` step 1 enumerates every project under
`~/.claude/projects/` before the ScopePaths filter drops most of
them. On a user with many historical projects this is wasted work —
scoping should narrow enumeration, not just post-filter it. Today
scope still helps because the per-project session-scan time is the
expensive part, not enumeration. Worth a 30-min optimization later
if scope-from-monorepo becomes a hot path.

**Forcing-function honesty:** F1 was a real CRITICAL bug — `shhh
audit .` silently disabled the user's flag intent, AND `shhh install
<path> --scope global` silently created junk directories. Both shipped
in the current session's "tested + green" patches. Caught only by
running the binary end-to-end against a real shell. This is exactly
why `make test` alone isn't enough — the bug was in argv parsing,
which tests don't exercise the same way real shell invocations do.

The fix + tests close the regression. The remaining items (F2–F6)
are quality-of-life and can wait until a user hits one.
